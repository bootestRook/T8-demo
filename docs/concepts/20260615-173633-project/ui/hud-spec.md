# UI 与 HUD

真实游戏首版需要基础 UI 闭环。没有 PSD、PNG 或 UI 源图时，AI 也应生成 UI sheet、HUD 图标、按钮、面板、进度条或状态徽章之一，并接入 `assets/ui/`。

## HUD 信息

- 空脚手架默认展示编辑器可见的 HUD 骨架：顶部目标、状态、提示；中心启动提示；底部下一步。
- 新游戏确认后，AI 必须把这些占位信息替换为当前玩法的真实目标、关键状态、进度/资源和阶段反馈。
- 顶部 HUD 必须能在窄屏重排或压缩长文本；目标、状态、提示不能依赖单行横排硬挤。
- 中心大提示只用于启动、暂停、阶段结算或明确决策；进入实时玩法后必须隐藏、降级到底部提示或换成不遮挡核心操作区的小型状态。
- 面板宽度必须受 viewport 约束，禁止用超过 360px 的固定最小宽度作为唯一布局策略。

## 菜单与结算

- 空脚手架不预置暂停、设置或复杂结算菜单。
- 首版玩法需要重开、下一关、升级、购买或选择时，应新增稳定 UI 场景或面板节点，并保留键盘/手柄焦点路径。

## UI 素材

- 空脚手架 HUD 使用程序化文字和面板作为编辑器可见占位。
- 正式首版未声明程序化占位时，至少生成或接入 HUD 图标、按钮、面板、进度条或状态徽章之一，放入 `assets/ui/` 并登记到 `docs/project/art/asset-manifest.json`。

## 验收

- 浏览器内能清楚看到目标、反馈、结算和下一步。
- 1280x720 和 390x844 截图下，HUD 不越界、不遮住核心玩法主体，长目标/状态文案不会挤出屏幕。
- `python scripts/visual_readability_review.py --strict` 必须通过 HUD 响应式布局检查；若涉及正式玩法，还需结合截图人工确认中心提示没有长期挡住操作区。
- UI 文案服务玩法，不解释工程结构。
